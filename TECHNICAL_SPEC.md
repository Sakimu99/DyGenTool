# 抖音文案提取与图片生成视频工具技术方案

## 1. 项目目标

构建一个可部署到 Cloudflare Pages 的 Web 工具，支持用户：

1. 粘贴一段抖音视频分享链接。
2. 自动提取该视频中的文本内容，主要用于获取视频文案。
3. 调用大语言模型对文案进行轻度改写，或生成类似风格的新文案。
4. 上传用户自己的图片和背景素材。
5. 根据图片、背景和文案自动生成一段短视频。
6. 在页面中预览并下载生成的视频。

## 2. 关键结论

当前阶段的用户规模假设是前期试水为主，用户基数不大，预计最多约 100 人。技术方案应优先保证低成本、低运维复杂度和快速验证；如果后期用户增长、渲染任务积压或稳定性要求提高，再迁移到更完整的生产架构。

### 2.1 推荐技术路线

推荐采用：

- 前端：React / Vite / TypeScript
- UI：Tailwind CSS + shadcn/ui 或 Radix UI
- 部署：Cloudflare Pages
- 后端 API：Cloudflare Pages Functions / Workers
- 文件存储：Cloudflare R2
- 元数据存储：Cloudflare D1 或 KV
- LLM：优先通过 Pages Functions 调用外部大模型 API；也可评估 Cloudflare Workers AI
- 视频生成：MVP 阶段优先采用浏览器端 `ffmpeg.wasm` 或 Canvas/WebCodecs 方案；生产级任务建议拆出独立视频渲染服务

### 2.2 重要限制

Cloudflare Pages / Workers 非常适合部署前端页面和轻量 API，但不适合直接承担重型视频渲染任务。

原因：

- Workers 运行时不支持传统 Node.js 长时间后台任务模式。
- 没有稳定的本地文件系统用于 FFmpeg 中间文件处理。
- CPU、内存、执行时间均受限制。
- 原生 FFmpeg 无法直接在 Workers 中运行。

因此，视频生成应分为两种实现路径：

1. **轻量 MVP：前端本地生成视频**
   - 用户浏览器负责图片合成、字幕绘制、音视频编码。
   - 服务端只负责文案提取、LLM 改写、临时存储。
   - 优点：部署简单，成本低，完全适配 Cloudflare Pages。
   - 缺点：依赖用户设备性能，手机端可能慢，复杂视频效果受限。

2. **生产级：Cloudflare + 外部视频渲染服务**
   - Cloudflare 负责前端、API、鉴权、存储和任务编排。
   - 视频渲染交给独立服务，例如 Fly.io、Render、Railway、Modal、Replicate、自建 GPU/CPU 服务等。
   - 如果必须采用低成本自托管方案，可以将树莓派 + Cloudflare Tunnel 作为独立渲染服务入口。
   - 优点：性能稳定，可控性强，支持复杂模板和批量任务。
   - 缺点：架构更复杂，成本更高；自托管方案还需要自行承担设备稳定性、散热、存储和运维风险。

## 3. 功能拆解

### 3.1 抖音链接解析与文案提取

用户输入示例：

```text
复制打开抖音，看看【xxx的视频】 https://v.douyin.com/xxxx/ ...
```

处理流程：

1. 前端从用户输入中提取 URL。
2. 后端请求短链，获取重定向后的真实地址。
3. 尝试从页面 HTML、结构化数据或接口返回中提取标题、描述、作者、视频 ID 等信息。
4. 返回文案候选内容给前端。
5. 前端允许用户手动编辑确认。

#### 风险与限制

抖音没有面向普通开发者稳定开放的视频文案解析 API，直接抓取页面存在以下问题：

- 页面结构可能频繁变化。
- 可能触发风控、验证码或反爬限制。
- Cloudflare Workers 的 fetch 环境不一定能稳定获取完整页面内容。
- 某些内容需要浏览器执行 JavaScript 后才可见。
- 需要注意平台服务条款与版权合规。

#### 推荐实现策略

MVP 阶段不要把“100% 自动解析抖音文案”作为强依赖，而是采用降级设计：

1. 优先自动解析分享链接中的 URL。
2. 后端尝试提取视频标题和描述。
3. 如果失败，让用户手动粘贴文案。
4. 前端始终提供“手动编辑文案”入口。

后续如果需要稳定解析，可选择：

- 接入合规第三方解析 API。
- 自建浏览器渲染解析服务，但不建议放在 Cloudflare Workers 中。
- 做浏览器插件，让用户在本机打开抖音页面时提取可见文本。

## 4. LLM 文案改写与仿写

### 4.1 功能模式

当前提供四种模式：

1. **保留原文**
   - 只清洗分享文本、平台描述、链接和其他噪声。
   - 不做风格改写。
   - 只返回一段候选文案。

2. **轻度改写**
   - 保留原意。
   - 调整表达方式。
   - 减少重复、口水化内容。

3. **同风格仿写**
   - 分析原文风格。
   - 生成同类短视频文案。
   - 保持节奏、语气、句式相似，但避免逐字复制。
   - 当前返回三段更适合视频口播的候选文案。

4. **营销增强**
   - 增强开头钩子。
   - 优化行动号召。
   - 适合商品、课程、服务推广场景。

### 4.2 推荐 API 设计

```http
POST /api/rewrite-copy
Content-Type: application/json

{
  "sourceText": "原始文案",
  "mode": "keep | rewrite | imitate | marketing",
  "tone": "自然 | 小红书风 | 口播风 | 情绪化 | 专业",
  "length": "short | medium | long"
}
```

返回：

```json
{
  "success": true,
  "cleanText": "清洗后的纯文案",
  "variants": ["备选文案 1", "备选文案 2"],
  "model": "deepseek-v4-flash"
}
```

`keep` 模式只返回一段候选；`imitate` 和 `marketing` 模式默认返回三段候选。

### 4.3 模型选择

可选方案：

| 方案 | 优点 | 缺点 | 推荐程度 |
| --- | --- | --- | --- |
| 外部大模型 API | 质量高、模型选择多 | 需要管理密钥和成本 | 推荐 |
| Cloudflare Workers AI | 和 Cloudflare 集成好 | 中文文案质量需实测 | 可评估 |
| 本地模型 | 成本可控、隐私好 | 部署复杂，不适合 Pages | 不推荐 MVP |

MVP 建议先通过 Pages Functions 调用外部大模型 API，并将 API Key 放在 Cloudflare 环境变量中，不暴露给前端。

当前文案生成默认使用 DeepSeek V4 Flash：`deepseek-v4-flash`。该场景主要是文案清洗、改写和仿写，不需要复杂推理，因此请求中显式关闭思考模式以降低延迟和成本。视频语音文本提取默认使用小米 MiMo：`mimo-v2.5`。

## 5. 图片上传与素材管理

### 5.1 上传内容

用户可以上传：

- 多张主体图片。
- 背景图片。
- 可选背景音乐。
- 可选 Logo 或水印。

### 5.2 存储方案

推荐使用 Cloudflare R2：

- 前端请求后端创建上传 URL。
- 前端直传图片到 R2。
- 后端保存素材元数据。
- 生成视频时读取这些素材。

MVP 如果采用浏览器本地视频生成，也可以不上传图片到服务端，直接使用浏览器 `File` 对象处理。

### 5.3 推荐策略

MVP 阶段：

- 图片直接在浏览器本地处理。
- 只在用户需要保存项目、异步渲染或跨设备访问时再上传 R2。

生产阶段：

- 图片上传至 R2。
- 任务信息存 D1。
- 视频生成结果也存 R2。
- 给用户提供短期有效下载链接。

## 6. 视频生成方案判断

### 6.1 方案 A：浏览器端生成视频

可选技术：

- Canvas 2D：绘制背景、图片、文字字幕。
- WebCodecs：更现代的浏览器编码能力，但兼容性需要处理。
- ffmpeg.wasm：在浏览器内运行 FFmpeg WebAssembly。
- MediaRecorder：录制 Canvas 动画流生成 WebM。

推荐 MVP 路线：

1. 使用 Canvas 生成视频画面。
2. 用 `canvas.captureStream()` + `MediaRecorder` 生成 WebM。
3. 如果必须导出 MP4，再评估 `ffmpeg.wasm` 转码。

优点：

- 完全适配 Cloudflare Pages。
- 不需要后端长任务。
- 成本低。
- 适合早期验证产品价值。

缺点：

- 移动端性能不稳定。
- MP4 导出支持复杂。
- 字体、字幕、转场效果需要自行实现。
- 大图、多图、长视频容易卡顿。

适合场景：

- 15-60 秒短视频。
- 图片轮播 + 字幕 + 简单转场。
- 用户量早期验证。

### 6.2 方案 B：外部服务端渲染视频

可选技术：

- Node.js + FFmpeg
- Remotion
- MoviePy
- Python + FFmpeg
- Headless Chrome + Canvas/WebGL 渲染

可选部署方式：

- 托管平台：Fly.io、Render、Railway、Modal 等，适合减少运维成本。
- 自托管设备：树莓派 + Cloudflare Tunnel，适合早期低成本验证、个人使用或约 100 人以内的小流量试水场景。

架构：

1. 用户上传图片到 R2。
2. Pages Functions 创建渲染任务。
3. 任务入队列。
4. 外部渲染服务拉取任务和素材。
5. 渲染服务生成 MP4。
6. 上传结果到 R2。
7. 前端轮询任务状态并下载。

如果采用树莓派 + Cloudflare Tunnel：

1. 树莓派运行独立渲染服务，只暴露经过 Tunnel 的 HTTPS 域名。
2. Pages Functions 创建任务后调用该渲染服务，或由树莓派定时拉取 D1/KV/队列中的待处理任务。
3. 树莓派下载 R2 素材，在本地使用 FFmpeg / Remotion / Python 生成 MP4。
4. 渲染完成后上传结果到 R2，并更新任务状态。
5. 前端仍只访问 Cloudflare Pages，不直接依赖树莓派内网地址。

优点：

- 输出 MP4 稳定。
- 支持复杂字幕、转场、动效、音频。
- 不依赖用户设备性能。
- 更适合商业化。

缺点：

- 需要额外服务部署。
- 成本更高。
- 需要处理任务队列、失败重试、资源清理。
- 树莓派方案受 CPU、内存、存储、散热和家庭网络稳定性影响，不适合作为高并发公共服务的唯一渲染节点。

适合场景：

- 对生成质量和稳定性要求较高。
- 用户设备不可控。
- 需要批量生成视频。
- 需要复杂模板。

### 6.3 最终建议

建议分阶段实现：

- **MVP：浏览器端生成 WebM/MP4，验证用户流程。**
- **增强版：接入 R2 + D1，支持保存项目。**
- **小流量自托管版：当用户规模约 100 人以内且任务量可控时，由树莓派 + Cloudflare Tunnel 承担独立渲染服务。**
- **生产版：当用户增长、排队时间变长或稳定性要求提高时，引入托管外部渲染服务或多节点自托管渲染服务，稳定生成 MP4。**

## 7. 推荐系统架构

### 7.1 MVP 架构

```text
用户浏览器
  ├─ 输入抖音链接
  ├─ 上传本地图片
  ├─ Canvas 生成画面
  ├─ MediaRecorder / ffmpeg.wasm 导出视频
  └─ 下载视频

Cloudflare Pages
  ├─ 静态前端
  └─ Pages Functions
       ├─ 解析抖音链接
       └─ 调用 LLM 改写文案
```

### 7.2 生产架构

```text
用户浏览器
  ├─ 输入链接 / 上传素材 / 选择模板
  └─ 查询任务状态 / 下载视频

Cloudflare Pages + Functions
  ├─ API 网关
  ├─ 鉴权
  ├─ 创建上传 URL
  ├─ 调用 LLM
  ├─ 创建视频生成任务
  └─ 查询任务状态

Cloudflare R2
  ├─ 原始图片
  ├─ 背景素材
  ├─ 音频素材
  └─ 生成视频

Cloudflare D1 / KV
  ├─ 用户项目
  ├─ 素材元数据
  ├─ 任务状态
  └─ 文案版本

外部视频渲染服务
  ├─ 托管服务或树莓派 + Cloudflare Tunnel
  ├─ 拉取素材
  ├─ 合成视频
  ├─ 上传 MP4 到 R2
  └─ 更新任务状态
```

## 8. 页面与交互设计

### 8.1 页面流程

建议做成单页分步向导：

1. **输入链接**
   - 粘贴抖音分享文本。
   - 自动识别 URL。
   - 显示解析结果。
   - 允许手动编辑文案。

2. **生成文案**
   - 选择改写模式。
   - 选择语气和长度。
   - 生成多个候选文案。
   - 用户选择最终文案。

3. **上传素材**
   - 上传主体图片。
   - 上传或选择背景。
   - 可选上传音乐。
   - 调整图片顺序。

4. **视频配置**
   - 选择比例：9:16、1:1、16:9。
   - 设置时长。
   - 设置字幕位置。
   - 设置转场样式。
   - 设置背景音乐音量。

5. **生成与下载**
   - 预览视频。
   - 生成文件。
   - 下载 WebM 或 MP4。

### 8.2 MVP 必须保留的兜底能力

- 文案解析失败时，允许手动粘贴文案。
- LLM 调用失败时，保留原始文案可继续制作。
- 视频生成失败时，提示降低图片数量、缩短时长或更换浏览器。
- 图片过大时，前端自动压缩。

## 9. API 草案

### 9.1 解析链接

```http
POST /api/extract-douyin
Content-Type: application/json

{
  "shareText": "用户粘贴的完整分享文本"
}
```

返回：

```json
{
  "success": true,
  "url": "提取出的分享链接",
  "pageUrl": "重定向后的页面 URL",
  "videoUrl": "识别到的视频播放地址",
  "description": "页面描述",
  "transcript": "MiMo 提取的视频语音文本"
}
```

如果只能提取页面描述但没有找到公开视频地址，应返回失败状态，并明确提示当前结果不是视频语音文本。

### 9.2 文案生成

```http
POST /api/rewrite-copy
Content-Type: application/json

{
  "sourceText": "原始文案",
  "mode": "rewrite",
  "tone": "口播风",
  "length": "medium"
}
```

返回：

```json
{
  "success": true,
  "cleanText": "清洗后的纯文案",
  "variants": ["候选文案 1", "候选文案 2"],
  "model": "deepseek-v4-flash"
}
```

`keep` 模式只返回一段候选；`imitate` 和 `marketing` 模式默认返回三段候选。未配置 `DEEPSEEK_API_KEY` 时，后端使用本地规则兜底生成候选并返回提示信息。

### 9.3 创建上传链接（生产阶段）

```http
POST /api/uploads/create-url
Content-Type: application/json

{
  "filename": "image.png",
  "contentType": "image/png"
}
```

### 9.4 创建渲染任务（生产阶段）

```http
POST /api/render-jobs
Content-Type: application/json

{
  "copyText": "最终文案",
  "assets": ["r2-key-1", "r2-key-2"],
  "background": "r2-background-key",
  "options": {
    "ratio": "9:16",
    "duration": 30,

  }
```

## 10. 技术选型建议

### 10.1 前端

推荐：

- React + Vite + TypeScript
- Tailwind CSS
- shadcn/ui
- Zustand 或 React Context 管理页面状态
- react-dropzone 处理文件上传
- browser-image-compression 压缩图片

原因：

- 开发效率高。
- Cloudflare Pages 部署成熟。
- 生态完整。
- 适合做多步骤交互工具。

### 10.2 后端

推荐：

- Cloudflare Pages Functions
- Hono 作为轻量路由框架
- Zod 做请求参数校验

原因：

- 与 Pages 集成自然。
- 适合轻量 API。
- 类型安全较好。

### 10.3 存储

MVP：

- 不落库，浏览器本地处理。
- 可用 localStorage 暂存配置。

生产版：

- R2：存图片、音频和生成视频。
- D1：存项目、任务和用户记录。
- KV：存短期缓存，例如解析结果和任务状态快照。

### 10.4 视频生成

MVP 推荐顺序：

1. Canvas + MediaRecorder 导出 WebM。
2. ffmpeg.wasm 转 MP4，作为增强能力。
3. 后续再引入服务端 FFmpeg / Remotion。

生产版推荐：

- 如果偏前端模板和动效：Remotion。
- 如果偏稳定批处理和简单合成：Node/Python + FFmpeg。
- 如果采用树莓派 + Cloudflare Tunnel，优先选择 FFmpeg 或轻量 Node/Python 渲染流程，避免过重的浏览器渲染依赖。
- 如果未来有 AI 视频能力：接入第三方生成式视频 API。

## 11. 合规与安全

需要注意：

1. 不应绕过抖音反爬、登录、验证码或权限控制。
2. 不应下载或搬运他人视频内容作为素材，除非用户拥有授权。
3. 文案仿写应避免逐字复制，尽量做风格参考和原创表达。
4. 用户上传文件需要限制类型和大小。
5. API Key 必须放在 Cloudflare 环境变量中。
6. R2 下载链接应设置有效期。
7. 如果开放公网使用，需要加入速率限制和成本控制。
8. 如果使用 Cloudflare Tunnel 暴露树莓派渲染服务，渲染接口必须增加服务端鉴权、签名或内网访问控制，不能直接向公网匿名开放。

## 12. 开发里程碑

### 阶段 1：MVP 验证

目标：跑通完整用户流程。

范围：

- 创建 React 页面。
- 粘贴分享文本并提取 URL。
- 尝试解析文案，失败时手动输入。
- 接入 LLM 生成改写文案。
- 上传多张本地图片。
- Canvas 生成竖屏视频画面。
- MediaRecorder 导出 WebM。
- 支持下载。

不做：

- 用户系统。
- 服务端视频渲染。
- 长期存储项目。
- 复杂模板市场。

### 阶段 2：体验增强

范围：

- 图片排序。
- 字幕样式配置。
- 多文案候选。
- 图片自动裁剪与适配。
- 背景音乐。
- ffmpeg.wasm 转 MP4。
- 前端性能优化。

### 阶段 3：小流量试水版

目标：在约 100 人以内的小规模用户场景下，验证服务端渲染、素材存储和异步任务是否有必要。

范围：

- R2 存储素材和成品。
- D1 存任务和项目。
- 树莓派 + Cloudflare Tunnel 承担独立渲染服务。
- 基础任务队列和失败提示。
- 简单鉴权、调用签名和并发限制。
- 记录渲染耗时、失败率和排队时长。

### 阶段 4：生产化

触发条件：用户量增长、渲染任务明显排队、树莓派稳定性不足，或需要更高 SLA。

范围：

- 迁移到托管外部渲染服务或多节点自托管渲染服务。
- 更完整的任务队列和失败重试。
- 鉴权和用户配额。
- 成本监控。
- 服务健康检查和告警。

## 14. 实现难点与解决建议

### 14.1 抖音文案解析不稳定

解决建议：

- 自动解析只是增强能力，不作为唯一入口。
- 一定保留手动粘贴文案。
- 将解析结果展示给用户确认。
- 对解析失败做清晰提示。

### 14.2 浏览器端视频生成性能

解决建议：

- 限制图片数量，例如首版最多 10 张。
- 限制输出时长，例如首版最多 60 秒。
- 上传后自动压缩图片。
- 首版优先导出 WebM。
- MP4 转码作为可选增强。

### 14.3 字幕排版

解决建议：

- 先做固定样式。
- 支持自动换行。
- 限制每屏字数。
- 长文案按句号、逗号、换行拆分。

### 14.4 成本控制

解决建议：

- 限制文案生成次数。
- 对链接解析结果做缓存。
- 控制单次上传大小。
- 生产版增加登录和额度。

## 15. 最小可行产品范围建议

首版建议只做以下能力：

1. 粘贴抖音分享文本。
2. 自动提取 URL，并展示重定向后的页面 URL。
3. 从抖音页面 HTML 中尝试提取公开视频播放地址，并展示识别到的视频地址。
4. 调用 MiMo 视频理解接口提取视频语音文本；失败时保留手动输入入口。
5. 支持原始文案清洗、保留原文、轻度改写、同风格仿写和营销增强。
6. 调用 DeepSeek V4 Flash 生成候选文案，仿写/营销模式返回三段候选。
7. 页面中的页面 URL、视频地址、原始文案、候选文案和最终文案支持一键复制。
8. 上传 1-10 张图片。
9. 选择 9:16、1:1、16:9 比例，设置时长和字幕位置。
10. 生成图片轮播 + 字幕视频。
11. 导出 WebM 下载。

这样可以最快验证核心价值：用户是否愿意用“已有爆款文案风格 + 自己图片”快速生成短视频。

## 16. 不建议首版实现的内容

首版不建议做：

- 自动下载抖音原视频。
- 绕过平台限制解析私密内容。
- 复杂 AI 视频生成。
- 多用户协作。
- 模板市场。
- 服务端批量渲染。
- 完整素材库。

这些功能会显著增加技术复杂度和合规风险，不符合 MVP 的 KISS 与 YAGNI 原则。

10. 已实现文案清洗与候选生成接口：
    - `keep`：保留原文，只返回一段。
    - `rewrite`：轻度改写，返回两段。
    - `imitate`：同风格仿写，返回三段更适合视频口播的候选。

12. 已在 DeepSeek 请求中显式关闭思考模式，降低普通文案生成的延迟和成本。
13. 已实现未配置 `DEEPSEEK_API_KEY` 时的本地规则兜底候选生成。
14. 已实现本地图片上传预览，首版限制最多 10 张图片。
15. 已实现 Canvas 视频画面绘制，支持 9:16、1:1、16:9 比例。
16. 已实现字幕位置配置：顶部、中部、底部。

## 18. 下一步建议

建议下一步围绕“视频效果差异化”继续增强，而不是过早进入复杂后端架构：

1. 增强字幕样式：描边、阴影、高亮词、分句展示。
2. 增加图片排序、裁剪、缩放和简单转场。
3. 增加语音合成能力，让最终文案可以直接生成旁白。
4. 评估数字人能力，但建议作为独立模块接入第三方服务。
5. 增加基础用量限制和错误日志，控制模型调用成本。
6. 继续观察浏览器端 WebM 生成是否能满足前期试水需求。
7. 当浏览器端导出质量、速度或兼容性成为瓶颈时，再引入树莓派 + Cloudflare Tunnel 的小流量服务端渲染方案。

在 MVP 跑通并确认用户确实需要保存项目、异步渲染或跨设备访问后，再判断是否引入 R2、D1 和外部视频渲染服务。
